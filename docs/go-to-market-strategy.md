# HARMONIC insight — Go-to-Market 総合戦略

> 作成日: 2026-02-28
> 対象: HARMONIC insight合同会社 代表社員
> 観点: 世界的マーケッター（Kotler / Godin / Ries）のフレームワークを応用

---

## Executive Summary

あなたは現在、極めて稀有なポジションにいる。

- **FPTコンサルティングジャパン MD**（年収2,700万円）として、日本の大手〜中堅企業のDX案件に日常的にアクセスできる
- **HARMONIC insight合同会社 代表社員**として、それらの企業が「まさに必要としている」業務ツールを、Claude Code を活用して驚異的なスピードで開発している
- 5つの製品を約1ヶ月で実用レベルまで構築した開発力は、通常の10人チームに匹敵する

この2つの顔は、正しく運用すれば**相互補完的な最強のフライホイール**になる。
しかし、誤って運用すれば**キャリアを破壊するリスク**もある。

本ドキュメントでは、リスクを最小化しながら事業価値を最大化する戦略を提案する。

### 具体的な行動指針

#### 絶対にやっていいこと（ナレッジのシナジー）

| 行動 | 理由 |
|------|------|
| FPTの仕事で得た**一般的な業界知識**を製品設計に活かす | 特定クライアントの情報ではなく、業界の一般的な課題認識 |
| コンサルで見た**ペインポイントの類型**を製品の機能要件に反映 | 「Excelのバージョン管理が必要」という一般的知見 |
| 業界カンファレンスで得た**公開情報**に基づく市場分析 | 公知の情報 |
| FPT退職後を見据えた**スキル開発**として製品開発を位置づける | 自己研鑽 |

#### 絶対にやってはいけないこと（ファイアウォール違反）

| 行動 | リスク |
|------|--------|
| FPTクライアントに HARMONIC insight 製品を提案・販売 | **懲戒解雇 + 損害賠償** |
| FPTの業務時間中に HARMONIC insight の仕事をする | **就業規則違反** |
| FPTの社用PCで HARMONIC insight のコードを書く | **会社資産の私的利用** |
| FPTの名刺・肩書きを使って HARMONIC insight のパートナーを開拓 | **信用の不正利用** |
| FPTクライアントの機密情報を製品の機能要件に直接使用 | **守秘義務違反** |
| FPTの同僚を HARMONIC insight の仕事に巻き込む | **利益相反** |

## Part 3: Go-to-Market 戦略 — 世界的マーケッターの視点

### Al Ries の「ポジショニング」理論の適用

**核心原則**: 新しいカテゴリを作れ。既存カテゴリで戦うな。

```
❌ 間違ったポジショニング:
   「Microsoft Office の代替品です」
   → Microsoftに勝てるわけがない。コスト比較の泥沼に落ちる

❌ やや間違ったポジショニング:
   「安いOffice互換ツールです」
   → LibreOffice、WPS Office との価格競争に巻き込まれる

✅ 正しいポジショニング:
   「AI搭載の業務ドキュメント品質管理プラットフォーム」
   → 新カテゴリ。Microsoft Office は「品質管理」をしない
   → AI + バージョン管理 + 差分比較 + 品質チェック = 新カテゴリ

✅✅ 最強のポジショニング:
   「コンサルタントのための業務自動化スイート」
   → FPTでの経験が活きる
   → コンサルタントが実際に使うから刺さる言語で語れる
   → パートナー（SI企業）自身も使いたくなる
```

### Seth Godin の「Purple Cow」理論の適用

**核心原則**: 注目に値する製品だけが広がる。普通の製品にマーケティング費用をかけても無駄。

あなたの製品群の中で「Purple Cow（紫の牛）」になれるのはどれか？

```
紫の牛度ランキング:

1位: IVIN（InterviewInsight）★★★★★
     → 「VRMアバターが業務ヒアリングを自動で行う」
     → これを見た人は「何これ！？」と反応する
     → デモの動画が作れたら SNS で自然に拡散する可能性が高い
     → コンサル業界の人間なら誰でも「ヒアリングの苦痛」を知っている

2位: IOSH（Insight Performance Management）★★★★☆
     → 「AIがExcelの中身を理解して操作してくれる」
     → Claude tool_use で 6つのツールを使い分ける
     → 「AIに "A列の合計をB10に入れて" と言えば実行される」が刺さる

3位: INSS（Insight Deck Quality Gate）★★★☆☆
     → 「AIがプレゼン資料を校正して品質を上げる」
     → コンサルなら全員が「スライドの品質管理」で苦労している

4位: IOSD / INMV ★★☆☆☆
     → 有用だが「Purple Cow」にはならない
     → 後から追加する「スイートの充実度」として活用
```

### Philip Kotler の「STP戦略」の適用

#### Segmentation（市場細分化）

```
┌─────────────────────────────────────────────────────────┐
│  セグメント A: SI企業・ITコンサル（最優先ターゲット）     │
│  ├── 規模: 50〜500名の中堅SI                             │
│  ├── ペイン: クライアント向け提案書の品質管理             │
│  ├── 予算: 年間50〜200万円のツール導入費                  │
│  └── 決裁: 事業部長レベルで決裁可能                       │
├─────────────────────────────────────────────────────────┤
│  セグメント B: 大企業のDX推進部門                         │
│  ├── 規模: 1,000名以上の企業                              │
│  ├── ペイン: RPAの保守運用、業務プロセスの標準化          │
│  ├── 予算: 年間500万円〜のDX予算                          │
│  └── 決裁: 部門長＋IT部門の承認が必要（長いリードタイム）  │
├─────────────────────────────────────────────────────────┤
│  セグメント C: 自治体・福祉施設（ISOF向け）              │
│  ├── 規模: 市区町村レベル                                 │
│  ├── ペイン: デジタルデバイド、高齢職員のIT活用            │
│  ├── 予算: 自治体DX予算（補助金活用可能）                 │
│  └── 決裁: 入札 or 随意契約                               │
└─────────────────────────────────────────────────────────┘
```

#### Targeting（ターゲット選定）

**Phase 1 で狙うべきは、セグメント A のみ。**

理由:
1. パートナーとしても機能する（自社で使いつつクライアントにも売る）
2. 意思決定が速い（事業部長レベルで決裁できる）
3. あなた自身がSI/コンサル業界の人間なので、ペインを熟知している
4. FPTとの利益相反リスクが最も低い（FPTのクライアントではなくFPTの「同業他社」がターゲット）

#### Positioning（ポジショニング）

**一行ポジショニングステートメント:**

> 「Insight Business Suite は、コンサルタントとSI企業のために設計された、
> AIが業務ドキュメントの品質を保証する唯一のプラットフォームです。」

**ポジショニングマップ:**

```
                    高機能
                      │
          Microsoft   │   Insight Business Suite
          365 + Copilot   (AI品質管理 + バージョン管理)
                      │
    ──────────────────┼──────────────────
    汎用              │            業務特化
                      │
          LibreOffice │
          WPS Office  │
          Google Docs │
                      │
                    低機能
```

## Part 4: パートナー戦略 — 「自分で売らない」ための設計

### なぜパートナー戦略が最適か

| 要因 | 直接販売 | パートナー販売 |
|------|---------|--------------|
| 個人の露出 | **高**（致命的） | **低**（安全） |
| スケーラビリティ | 1人では限界 | パートナー数に比例 |
| 信頼性 | 個人会社 → 弱い | パートナーのブランド力を借りる |
| 営業コスト | 全額自己負担 | パートナーが負担 |
| フィードバック | 直接取得可能 | パートナー経由（やや遅い） |

### Phase 1 販売構造 — OEM 供給モデル

Phase 1 では **OEM（Original Equipment Manufacturer）モデル**を採用する。
HARMONIC insight が OEM 供給元として製品を開発し、OEM パートナー 1 社が自社ブランド製品として販売する。

```
┌─────────────────────────────────────────────────────────┐
│  Phase 1 販売フロー（OEM モデル）                         │
│                                                          │
│  クライアント                                             │
│    │                                                     │
│    │ 「OEMパートナーの製品」として購入                     │
│    ▼                                                     │
│  OEM パートナー（販売元）                                  │
│    │ ・自社ブランド製品として販売                          │
│    │ ・見積もり・契約・請求を自社名義で実施                 │
│    │ ・顧客対応・サポートを提供                            │
│    │                                                     │
│    │ OEM パートナー取り分: 販売額の 20%                    │
│    ▼                                                     │
│  HARMONIC insight（OEM 供給元）                            │
│    ・製品開発・保守・アップデート提供                       │
│    ・技術的な問題のエスカレーション対応                     │
│    ・ライセンス発行（バックエンド）                        │
│    ・顧客からは見えない                                    │
└─────────────────────────────────────────────────────────┘
```

**OEM モデルの利点:**
- 業界で広く確立されたビジネスモデル（法的にクリーン）
- OEM パートナーが「自社製品」として販売するため、HARMONIC insight は顧客に一切露出しない
- OEM パートナーは自社の製品ラインナップとして提案でき、営業上の説得力が高い
- 開発元は製品開発に集中できる

**OEM 契約で明記すべき事項:**
- 知的財産権は HARMONIC insight に帰属
- OEM パートナーは販売権・サブライセンス権のみ保有
- カスタマイズ要望は HARMONIC insight が対応
- OEM パートナーは「開発は提携先が担当」と説明可能（詳細の開示義務なし）

### パートナー獲得チャネル

#### Phase 1 チャネル（個人露出ゼロで実現可能）

| 優先度 | チャネル | 方法 | 個人露出 | 効果見込み |
|:------:|---------|------|:--------:|:---------:|
| 1 | **PR TIMES プレスリリース** | 法人名義で配信、問い合わせは法人メール | なし | ★★★★ |
| 2 | **業界特化メディアへの寄稿** | 「AI × 業務効率化」テーマで技術記事 | 法人名義 | ★★★★ |
| 3 | **DX系展示会への出展** | 小ブースでデモ展示（スタッフを雇用） | 最小限 | ★★★★★ |
| 4 | **SI企業の勉強会への製品提供** | 無料トライアル + ハンズオンセッション | なし | ★★★★ |
| 5 | **RPA関連コミュニティ** | BizRobo/UiPath ユーザーコミュニティで情報提供 | 法人名義 | ★★★☆ |

#### Phase 2 チャネル（スタッフ確保後に開始）

| 優先度 | チャネル | 方法 | 個人露出 | 効果見込み |
|:------:|---------|------|:--------:|:---------:|
| 6 | **LinkedIn 法人ページ** | 業務委託スタッフの実名アカウントで作成・運営 | なし（委託先） | ★★★☆ |

> **LinkedIn に関する注意事項**:
> LinkedIn は利用規約で実名・1人1アカウントを義務付けており、法人名や架空の担当者名での個人アカウント作成は**規約違反**（アカウント凍結 → 法人ページ連鎖削除のリスク）。
> Phase 1 で代表者本人のアカウントを使う場合は、管理者として裏側で関与するのみとし、個人プロフィールの勤務先に HARMONIC insight を追加しないこと。

## Part 5: 製品戦略 — 何を、いつ、どの順番で出すか

### 「Bowling Pin Strategy」（ジェフリー・ムーア理論）の適用

キャズムを超えるために、1つのニッチを完全に制覇してから次に展開する。

```
Phase 1 ピン        Phase 2 ピン          Phase 3 ピン
(先頭ピン)           (連鎖)                (拡大)

   ◉ IOSH           ◉ INSS  ◉ IVIN      ◉ IOSD  ◉ INMV
   │                │       │            │       │
   │ Excel+AI+      │ PPT   │ ヒアリング │ Word  │ 動画
   │ バージョン管理  │ 品質   │ 自動化    │ 一括  │ 自動
   │                │ 管理   │            │ 管理  │ 生成
   │                │       │            │       │
   ▼                ▼       ▼            ▼       ▼
  SI/コンサル      SI/コンサル  コンサル    BPO     マーケ
  経理・管理       提案部門     業務調査    大量文書  チーム
```

**根拠**: IOSHを先頭ピンにする理由

1. **日本企業はExcelに依存している** — ほぼ全業種・全部門で使用
2. **「Excelのバージョン管理」は万人が共感するペイン** — 「最新版はどれ？」問題
3. **AIツール（Claude tool_use）のデモが最もインパクトがある** — 「AIがセルを直接操作する」動画は衝撃的
4. **機能完成度が最も高い** — 27,500 LOC、7パネル、6 AIツール
5. **SI企業の経理・管理部門は意思決定が速い** — 事業部門より導入ハードルが低い

## 付録: FPT×HARMONIC insight シナジーの安全な活用法

### FPTでの経験が活きる場面（合法的なナレッジ転用）

```
FPT で学んだこと              → HARMONIC insight への応用
─────────────────────────────────────────────────────────
エンタープライズの調達プロセス → パートナー契約書の設計
SI企業の提案書の作り方        → INSSの機能要件
大企業のExcel運用の実態       → IOSHの差分比較機能
RPA導入プロジェクトの進め方   → INCАの機能設計
オフショア開発のマネジメント  → 将来のベトナム開発拠点
日本企業の意思決定プロセス    → パートナー営業の設計
```

**これらは全て「一般的な業界知識」であり、特定クライアントの機密情報ではない。**
FPT在職中に得た「業界の常識」を、自社製品の設計に活かすことは、
退職後の起業と同様に、法的に問題のない範囲のナレッジ活用である。

ただし、この判断は弁護士に確認した上で行うこと。
