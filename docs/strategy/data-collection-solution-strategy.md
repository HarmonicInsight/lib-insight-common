## 2. StravisLINKの設計思想 ── なぜ10年経っても通用するか

### 2-1. 設計者だからこそ知っている本質

StravisLINKを設計・開発した10年の経験から得た、データ収集の本質的な課題:

| # | 課題 | StravisLINKでの解決 | InsightSheetでの進化 |
|---|------|-------------------|--------------------------|
| 1 | 現場はExcelを変えたくない | Excelフォームをそのまま入力画面に | Excelネイティブなまま収集 |
| 2 | フォーマットが拠点ごとにバラバラ | 連携シートで吸収 | テーブル定義で自由にマッピング |
| 3 | 中央DBの運用コストが高い | クライアント/サーバー型 | クラウド/軽量DB対応 |
| 4 | 収集状況が見えない | ステータス管理機能 | リアルタイムダッシュボード |
| 5 | エラーデータが混入する | エラーチェック定義 | バリデーション＋AI検知 |
| 6 | 導入に時間がかかる | SI必須（数ヶ月〜1年） | セルフ導入可能（数週間） |

### 2-2. StravisLINKのデータ収集アーキテクチャ

StravisLINKは2つのデータ収集方式を持っていた:

```
StravisLINKのデータ収集アーキテクチャ:

  ① 固定テーブル方式（連結会計データ用）
  ─────────────────────────────────────
  ・BS/PL/CF等、収集データの構造が最初から決まっている
  ・科目体系に沿った専用DB列にマッピング
  ・特許対象の仕掛け（FS科目体系への入力パッケージ生成）

  ② 汎用テーブル方式（それ以外のデータ用）
  ─────────────────────────────────────
  ・1つの物理テーブルにN個の論理テーブルを格納
  ・レコードとしてテーブルを持つ構成
  ・テーブル構造を自由に定義可能
  ・連結会計に限らない任意のデータを収集
  ・RDBの制約内で柔軟性を実現する工夫
```

### 2-3. 汎用テーブルの思想を「AI対応JSONデータベース」に進化させる

StravisLINKの汎用テーブル（1物理テーブル→N論理テーブル）の思想を、
AI時代のアーキテクチャとして再設計する:

```
StravisLINK（旧）:
┌──────────────────────────────────────────────────────────┐
│  1つの物理テーブル                                        │
│    ├─ table_id: 論理テーブル識別                          │
│    ├─ col1, col2, ... colN: 汎用カラム                   │
│    └─ メタデータで列の意味を管理                          │
│                                                          │
│  制約: RDBの列数上限、型の制約、スキーマ変更にALTER TABLE  │
└──────────────────────────────────────────────────────────┘

InsightSheet（進化版）:
┌──────────────────────────────────────────────────────────┐
│  AI対応JSONデータベース                                    │
│    ├─ テーブル定義 = JSON Schema                          │
│    │    {"tableName": "月次報告",                          │
│    │     "columns": [                                     │
│    │       {"name": "拠点", "type": "string"},            │
│    │       {"name": "売上", "type": "number"},            │
│    │       {"name": "期間", "type": "date"}               │
│    │     ]}                                               │
│    │                                                      │
│    ├─ データ格納 = JSON                                   │
│    │    {"拠点": "大阪支店", "売上": 5000000,              │
│    │     "期間": "2026-01"}                                │
│    │                                                      │
│    ├─ バリデーション = JSON Schema Validation              │
│    └─ AI連携 = JSONをそのままLLMに渡せる                  │
│                                                          │
│  利点: スキーマ変更が即時、AIネイティブ、柔軟性が本質的    │
│  許容: パフォーマンスは多少落ちてもOK                      │
└──────────────────────────────────────────────────────────┘
```

#### なぜJSON型データベースか

| 観点 | StravisLINK（RDB汎用テーブル） | InsightSheet（JSON DB） |
|------|-------------------------------|------------------------------|
| スキーマ定義 | メタデータテーブル + ALTER TABLE | **JSON Schemaで定義 → そのまま格納定義** |
| スキーマ変更 | DB変更 + マイグレーション + テスト | **JSONスキーマを更新するだけ。即時反映** |
| AI連携 | データ変換・整形が必要 | **JSONをそのままAIに渡せる。AIが直接読める** |
| バリデーション | アプリ層で個別実装 | **JSON Schema Validationで標準化** |
| テーブル定義のUI | 専用管理画面が必要 | **JSONエディタ or AIが自動生成** |
| 柔軟性 | 論理テーブルの工夫で対応 | **ネイティブに柔軟。ネスト構造も自然** |
| パフォーマンス | RDB最適化済み。高速 | **多少落ちるが、データ収集用途なら十分** |
| 検索 | SQL | **JSONクエリ（PostgreSQL JSONB等）** |

#### AI対応が「次世代」を決定づける

```
従来のデータ収集:
  Excel → アプリで変換 → RDBに格納 → SQLで取得 → レポート作成

InsightSheetのデータ収集:
  Excel → JSON変換 → JSON DBに格納 → AIが直接読む → AIがレポート作成
                                         ↑
                                   ここが革命的

  「AIにデータを読ませたい」
  → JSONなら変換不要。そのまま渡せる
  → テーブル定義（JSON Schema）も一緒に渡せば、AIが構造を理解する
  → 「この月次報告データを分析して」と言うだけ
```

#### 技術選定の方向性

| 選択肢 | 特徴 | 適合度 |
|--------|------|--------|
| **PostgreSQL JSONB** | RDBの堅牢性 + JSONの柔軟性。インデックス可 | **最有力** ── 既存スキルセット活用＋JSON対応 |
| MongoDB | ネイティブJSON。スキーマレス | 候補 ── 純粋なドキュメントDB |
| SQLite + JSON1 | 軽量。組み込み可能 | 小規模導入向け |
| Supabase (PostgreSQL) | クラウドネイティブ。API自動生成 | クラウド版に最適 |

### 3-1. StravisLINKが解決してきた市場 vs 新たな市場

```
StravisLINKの市場（連結会計）:
  ・上場企業のグループ経営管理
  ・連結決算データの収集
  ・BEPS/CBCR等の税務情報収集
  ・予算管理
  → 顧客: 大手上場企業（導入実績950社超）
  → 価格帯: 数千万円〜

InsightSheetの市場（汎用データ収集）:
  ・あらゆる業界の「Excelで集めてるデータ」
  ・部署間・拠点間のデータ収集
  ・グループ会社からの定期報告
  ・プロジェクト横断のデータ統合
  → 顧客: 中堅企業〜大手（StravisLINKに手が出ない層）
  → 価格帯: 200万円〜
```

### 3-2. 「Excelで集めてる」問題の普遍性

日本企業の現実:

| 業務 | 現状 | 課題 |
|------|------|------|
| 予算収集 | 各部署がExcelで提出 → メールで集約 | バージョン管理が崩壊、集計に丸1日 |
| 月次報告 | 拠点ごとにExcelで報告 → 本社が手動集計 | フォーマット不統一、提出遅延 |
| 品質データ | 工場ごとにExcel管理 → 本社に送付 | エラー混入、最新版が不明 |
| 人事データ | 各事業所がExcelで人員報告 | 入力ミス、集計作業の属人化 |
| プロジェクト進捗 | PMがExcelで収集 → 経営層に報告 | リアルタイム性なし |
| ESG/サステナビリティ | 各拠点のCO2排出量等をExcelで収集 | 監査対応で収集の正確性が問われる |
| 安全衛生報告 | 現場からExcelで月次報告 | 提出漏れ、フォーマット不統一 |

**→ 全部「StravisLINKの汎用テーブル」の設計思想で解決できる課題**

### 4-3. なぜ200万円が「絶妙」か

| 視点 | 理由 |
|------|------|
| **エンタープライズの信頼感** | 200万円は「本格的なエンタープライズ製品」。100万円より格が上がる |
| **決裁ライン** | 中堅企業の部長〜役員決裁で通る金額帯。高すぎて通らないことはない |
| **利益率** | 原価はほぼ同じ。100万円の倍の売上が立つ |
| **値引き余地** | 「通常200万のところ、初期導入企業は150万で」という交渉カードが使える |
| **パートナー販売** | 卸値140万円。パートナーが導入支援込みで280万円で提案 → パートナー利益140万/案件 |
| **kintoneとの棲み分け** | kintoneは月額課金で安いがExcel感覚ではない。200万円の買い切りで明確な差別化 |
| **ROI訴求** | 「手作業の集計に月40時間 × 12ヶ月 = 480時間/年。時給3,000円で144万円。1.5年で回収」 |
| **それでも圧倒的に安い** | STRAVIS-LINK（400万＋本体600万）の **1/5**。導入総額比で **1/10以下** |

## 9. 差別化の核心 ── 「経験」が最大の武器（書籍第5章）

### 9-1. 競合が絶対に言えないこと

> **「StravisLINKを設計・開発した10年の経験を持つ人間が、
> 同じ課題に対してゼロから作った次世代のデータ収集ソリューション」**

これは：
- kintoneの開発者には言えない（データ収集専用設計ではない）
- Forguncyの開発者には言えない（エンタープライズデータ収集の現場を知らない）
- Power Appsの開発者には言えない（Excelの現場を本当には理解していない）

### 9-2. 差別化ステートメント

```
「StravisLINKは素晴らしい製品です。
 でも、400万円＋STRAVIS本体が必要です。

 私たちはStravisLINKの設計思想を知り尽くした上で、
 その核心だけを取り出し、200万円で提供します。

 現場はExcelのまま。
 本社はデータベース。
 それだけのことに、3,000万円は要りません。」
```

### 9-3. 訴求に使える権威性

| 権威性の要素 | 内容 |
|-------------|------|
| **設計経験** | エンタープライズ級データ収集システムを10年間設計・開発 |
| **導入知見** | 世界中の拠点からのデータ収集の現場を熟知 |
| **課題理解** | フォーマット不統一・提出遅延・エラー混入等のリアルな痛み |
| **技術判断** | 何が本当に必要で、何が過剰な機能かを見極められる |
